Quickstart

Once you have access, get COLT running in under five minutes.

  1. Install the COLT package using your access credentials
  2. Create a colt.yaml configuration file
  3. Run discovery on your application
  4. Execute your first task
# Install COLT (requires access credentials)
pip install colt-agent
playwright install chromium

# Create config
colt init
# Edit colt.yaml with your app's base URL and LLM provider

# Discover your application
colt discover --config colt.yaml

# Execute a task
colt execute --react "Update the billing address to 456 Oak St, Denver CO 80202"

Configuration

COLT uses a YAML configuration file. Run colt init to generate a starter config.

# colt.yaml
project:
  name: "my-app"
  base_url: "http://localhost:3000"
  output_dir: "./colt-output"

llm:
  provider: "groq"      # or "openai", "ollama"
  model: "llama-3.3-70b-versatile"
  api_key_env: "GROQ_API_KEY"

browser:
  headless: true

Note: The api_key_env field references an environment variable name, not the key itself. Set your key with: export GROQ_API_KEY="your-key"

Supported LLM Providers

Groq (fast, recommended for dev), OpenAI, Ollama (local, no API key), or any OpenAI-compatible API.

Discovery

The discover command crawls your target application, discovers all interactable elements via sensory.py, generates descriptions, and indexes them into ChromaDB.

colt discover --config colt.yaml

Generates page purpose documents and how-to documents as vector embeddings. Also runs 7 hidden UI discovery strategies: timed UI, contextual UI, keyboard shortcuts, context menus, scroll-loaded content, conditional UI, and lazy-loaded tabs.

Execution

Two modes: planner (static, single LLM call) and ReAct (autonomous, iterative).

# Planner mode (simple tasks)
colt execute "Update the billing address to 456 Oak St"

# ReAct mode (recommended for most tasks)
colt execute --react "Change the admin email to admin@newdomain.com"

ReAct Mode

Iterative Observe-Reason-Act loop. Handles unexpected modals, loading states, error conditions. Max 25 iterations, 30 LLM calls hard abort. 3,500-token observation budget per iteration. 1-2-3 step semantic loop detection prevents repetition.

Search

Query the semantic index directly without executing a task:

colt search "how to change password"

Returns ranked results with similarity scores, target URLs, and how-to context. Useful for debugging index quality.

COLT App

Generate production-ready agent code using your semantic index as context. The app resolves relevant pages and elements, then outputs a standalone Python script.

# Generate an agent from a natural language description
colt app generate "Monitor billing for overdue invoices and extract amounts"

# Output: agents/billing_monitor.py
# Includes: navigation, selector cascade, fallback chains, error handling

How context resolution works

Queries ChromaDB to identify which pages and elements are relevant. Resolves selectors from the discovery-time cascade. Maps navigation paths between pages. Generated code uses actual selectors from your application — not placeholders.

Exporting agents

Generated agents are standalone Python scripts using Playwright directly. No COLT runtime needed. Drop them into CI/CD, cron jobs, or import into your code:

# Run directly
python agents/billing_monitor.py

# Or import into your code
from agents.billing_monitor import monitor_overdue
result = monitor_overdue(config)

Architecture Overview

Five-phase pipeline: Discover, Index, Search, Execute, Verify. Three core modules: sensory.py, planner.py, and react.py (1,553 lines).

Sensory Engine

Discovers standard form elements (input, textarea, select), interactive elements (button, a[href], [onclick]), ARIA role elements, test identifiers ([data-testid]), and disclosure elements (summary). Visibility verified via getBoundingClientRect() plus CSS property checks.

Selector Cascade

Priority: #id (with CSS.escape) → [data-testid] → [name] → [aria-label] → [title]. All selectors use CSS.escape() for special characters. Shadow DOM uses >> piercing; iframes use >>> boundary notation.

Best Practices

Run discovery after significant UI changes to keep the semantic index fresh. Use ReAct mode for tasks involving modals, multi-step workflows, or dynamic UI. Planner mode works for simple, predictable tasks. Write specific, descriptive task instructions. For dev, Groq with llama-3.3-70b-versatile is the best speed-accuracy balance.

Troubleshooting

Discovery finds 0 elements

Ensure your app is running at base_url. Check Playwright can launch Chromium. Handle authentication/cookies if required.

ReAct loop exhausts iterations

Task description may be too vague, semantic index may be stale, or target UI may require unsupported interactions. Re-run discovery and use more specific descriptions.

Selectors fail to match

If your app uses dynamic IDs, add data-testid, name, or aria-label attributes to key elements. Re-run discovery after attribute changes.
